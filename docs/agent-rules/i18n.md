# i18n — переводы интерфейса (EN/RU)

Переводится **только обвязка интерфейса** (нав, кнопки, тултипы, заголовки колонок/секций, фильтры тулбара, блёрбы). Тело патч-нотов (имена героев/предметов, текст изменений от Valve) и токены-теги (BUFF/NERF/NEW/DEL/REWORK/MISC/QoL — единый цветной словарь, повторяется в теле тысячи раз) остаются на английском. Английский — это baseline; словарь хранит только не-дефолтные языки (`ru`).

Механизм — клиентский, целиком в `src/scripts.js` (рантайм встроен, без отдельного файла, чтобы переключение было мгновенным и не зависело от `fetch`):
- Переключатель `EN|RU` живёт в общей навигации (`site_common.render_top_nav`, класс `.lang-switch`), значит появляется на каждой странице из одного определения. Выбор — в `localStorage('sloppy_lang')`.
- Каждый переводимый элемент несёт `data-i18n="key"` (textContent), `data-i18n-html="key"` (innerHTML с разметкой) или атрибутный вариант `data-i18n-title` / `data-i18n-placeholder` / `data-i18n-aria-label="key"`.
- `applyLang(lang)` обходит DOM: для `en` восстанавливает захваченный baseline, иначе ставит перевод из словаря `DICT`. Срабатывает на старте и при каждом переключении (+ событие `sloppy:langchange`).
- Строки, которые JS строит в рантайме, используют `t('key', 'English fallback')` (читает текущий язык в момент рендера). Узлы, которые JS создаёт, тоже могут нести `data-i18n`, тогда их подхватит `applyLang`.

Как добавить перевод строки:
1. В билдере/`site_common.py` повесь `data-i18n="<namespace>.<key>"` на элемент (английский текст оставь как есть — это baseline). Если у элемента есть дочерние узлы (caret, sort-ind, input) — оберни именно текст в свой `<span data-i18n=...>`, иначе textContent-подмена их сотрёт. Многострочные ярлыки-аббревиатуры (`EHP\nphys`) не трогаем.
2. Добавь запись в объект `DICT` в `src/scripts.js`: `'<namespace>.<key>': { ru: '…' }`.
3. Пересобери страницу и проверь, что все использованные ключи есть в `DICT` (аудит: вытащить `data-i18n*="..."` из HTML и `t('...')` из JS, сверить с ключами `DICT`).

Namespaces: `nav.*`/`mat.*` (общая навигация), `idx.*` (хаб), `patch.*` (патч-страница), `cal.*` (календарь), `dyn.*` (матрицы Hero/Item Dynamics, общий рендер `dyn_matrix_common.py`), `mi.*` (Mana Items), `cr.*`/`ua.*` (Neutral Stats/Abilities), `hs.*` (Hero Stats), `terr.*` (Terrain), `hl.*` (Hero Lab, целиком на JS), `js.*`/`ui.*` (общие JS-строки). Страницы Silent Changes (`builders/silent.py`) НЕ переводятся — у них свой inline-стиль, нет общей навигации и не подключается `scripts.js`, так что переключателя/рантайма там нет.

## Тело патч-нотов (RU) — отдельный механизм

Текст самих изменений (строки `li(...)`) переводится **инлайново**, не через `DICT` (тело слишком большое для центрального словаря). Источник — официальный перевод Valve: `data/patchnotes_english.txt` ↔ `data/patchnotes_russian.txt`, два KV-файла с общим ключом (`DOTA_Patch_<ver>_<entity>`). `patch/patch_i18n.py` склеивает их по ключу в карту `english_text → russian_text` (~22k пар) и отдаёт `ru_for(text)`; `patch/elements.py::li()` ищет **исходный** текст строки (до трансформаций) и, если нашёл, вешает перевод на спан `row-text` как `data-i18n-ru="…"`. Бейджи/числа лежат в соседних узлах (вне `row-text`), поэтому при переключении на RU не теряются. Покрытие ~88%; рукописные/перефразированные строки просто остаются на английском. Структурные заголовки (`section()`, `plain_header()`, чипы фильтра «Group:») берут перевод из небольшого ручного словаря `_STRUCTURAL` в том же модуле (`ru_for_header`).

Рантайм (`applyLang` в `scripts.js`) обрабатывает `data-i18n-ru` отдельно от `DICT`: для `ru` ставит `textContent` = перевод, для остальных языков восстанавливает захваченный английский `innerHTML`. Источники KV **должны** быть закоммичены (CI собирает патч-страницы из них). Чтобы расширить покрытие — дополнить `_STRUCTURAL` или (для свежих патчей) обновить `data/patchnotes_*.txt`.
